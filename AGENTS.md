# AGENTS.md

## 1. Назначение

Документ задаёт обязательные правила для AI-агентов.

Цели изменений:
- читаемые тесты
- простая архитектура
- минимальная стоимость поддержки

Если изменение не улучшает хотя бы одну цель, его не вносим.

---

## 2. Архитектура

Обязательное разделение слоёв:

`tests -> pages/api -> core`

### Пакеты и ответственность

- `tests`
  Только тестовые сценарии и проверки.

- `pages`, `api`
  Page Objects и API-клиенты: только действия пользователя/клиента.

- `core`
  Драйвер, конфигурация, базовая инфраструктура.

### Правила по слоям

- Тесты не импортируют `core`.
- Тесты не содержат локаторы.
- Page Objects и API-клиенты не содержат assert (кроме `isOpened`/`isLoaded`).
- В `core` нет бизнес-логики.

---

## 3. Правила тестов

### Обязательно

- Тест описывает сценарий пользователя.
- Название теста отражает бизнес-смысл.
- Все assert находятся только в тестах.

### Запрещено

- `Thread.sleep`, `Selenide.sleep`
- `new WebDriver`
- `if`, `switch`
- циклы
- вычисления и ветвления бизнес-логики
- assert вне тестов

---

## 4. Правила Page Objects / API

### Разрешено

- действия (`click`, `type`, `select` и т.д.)
- навигация
- `isOpened` / `isLoaded`

### Запрещено

- assert
- бизнес-логика
- условия
- вычисления
- лишние абстракции

---

## 5. Конфигурация

### Разрешено

- `*.properties`

### Запрещено

- хардкод URL, credentials, селекторов
- конфиг в Java-коде

---

## 6. Антипаттерны (исправляются сразу)

- любой `sleep`
- ручное управление `WebDriver`
- XPath без необходимости
- God-классы
- универсальные методы с большим числом параметров
- логика в Page Object

---

## 7. Правила для AI-агента

### Агент обязан

- соблюдать архитектуру из раздела 2
- писать максимально простое решение
- снижать дублирование без новых слоёв

### Агенту запрещено

- добавлять новые абстракции без явной необходимости
- делать «умные» обёртки
- добавлять retry/магические механики
- вводить новые конфигурационные слои

---

## 8. Проверка результата перед завершением

Минимум, который должен быть выполнен перед сдачей:

1. Проект собирается: `mvn -q -DskipTests compile`
2. Формат и стиль проходят: `mvn checkstyle:check`
3. Тесты проходят: `mvn -q test`
4. Нет флаки-поведения на повторном запуске критичных тестов.
5. Нет неиспользуемого кода и мёртвых методов.
6. Изменённый код читается без дополнительных объяснений.

Если любой пункт не выполнен, изменение не готово.

---

## 9. Критерии готовности

Изменение принимается только если:

- тест читается как пользовательский сценарий
- нет хардкода
- нет `sleep`
- нет лишней сложности
- архитектура слоёв не нарушена
- все проверки из раздела 8 выполнены

---

## 10. Финальное правило

При сомнении выбирай самое простое решение без новых абстракций.
