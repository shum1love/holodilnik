# 🤖 AGENTS.md — Правила для AI и разработчиков

Этот файл — **главный закон** репозитория.
Нарушил — сразу рефакторим.

Если коротко:
**Тесты — про бизнес. Код — простой. Магии — ноль.**

---

## 🎯 Главный принцип

> Любое изменение должно делать тесты:
> - **читаемее**
> - **стабильнее**
> - **дешевле в поддержке**

Если не делает — не надо. Даже если «красиво».

---

## 🧠 Архитектура

Tests
↓
Pages / API Clients
↓
Framework Core
**Жёсткое разделение:**

- **Tests** → что проверяем (бизнес-сценарий)
- **Pages/Clients** → как взаимодействуем
- **Framework** → как это работает внутри

Смешивать нельзя. Никогда.

## 🧪 Тесты

### Делай так:

- Тест = понятный сценарий пользователя
- Название теста = бизнес-смысл
- Ассерты — только в тестах (кроме `isOpened()`/`isLoaded()`)
- Используй Page Objects

### Не делай:

- `Thread.sleep()` / `Selenide.sleep()`
- `new WebDriver()`
- логику в тестах (`if`, циклы, вычисления)
- проверки внутри Page Object

**Хороший пример:**

```java
@Test
void пользователь_может_добавить_товар_в_корзину() {
    mainPage.open();

    productPage = mainPage.openFirstProduct();
    productPage.addToCart();

    cartPage.shouldContainProduct(productName);
}
```
## 🧱 Page Objects

Только:

действия (click, type, select, addToCart и т.д.)
isOpened() / isLoaded()

Запрещено:

Бизнес-логика
Ассерты
if / условия
вычисления
наследование ради наследования

Плохо:
if (price > 1000) clickDiscount();
Нормально:
applyDiscountIfNeeded(); // логика — в тесте

🔧 Конфигурация
Всё в properties-файлах.
Разрешено:

selenide.properties
config.properties

Запрещено:

Хардкод URL, логинов, паролей, селекторов
тащить конфиг в код


🚫 Антипаттерны
Увидел — исправляй без вопросов:

Thread.sleep / Selenide.sleep
new WebDriver()
XPath (если можно CSS)
God-классы (Utils, BasePage на 100500 методов)
Логика внутри Page Object
Захардкоженные селекторы/URL/креды
Наследование «на всякий случай»
«Универсальные» методы с 10 параметрами


🤖 Правила специально для AI-агентов
Ты обязан:

Соблюдать архитектуру выше — без исключений.
Предлагать самое простое решение.
Улучшать читаемость кода.
Убирать дублирование.
Никогда не добавлять новую магию, абстракции и «умные» обёртки.
Если хочешь добавить что-то сложнее 20 строк — сначала спроси.

Категорически запрещено:

Предлагать новые универсальные UI-обёртки/ретраи/automatic Allure steps "на будущее"
Вводить новый конфиг-стек поверх текущего (оставаться в `application.conf` + `*.properties`)
Делать BasePage<T> с fluent interface
Выносить локаторы в дополнительные пакеты/слои без необходимости
Добавлять «полезные» утилиты «на будущее»


🏁 Definition of Done
Изменение принято, только если:

Тест читается как история пользователя
Нет хардкода
Нет sleep
Нет лишней магии
Код можно понять за 20–30 секунд
Архитектура не нарушена


🏆 Финальная мысль
Хороший тест — скучный тест.
Если код выглядит «умным» — он почти наверняка говно.
Делай просто. Делай правильно. Делай так, чтобы через 3 года любой джуниор мог поддерживать без боли.
Теперь ты знаешь правила.
